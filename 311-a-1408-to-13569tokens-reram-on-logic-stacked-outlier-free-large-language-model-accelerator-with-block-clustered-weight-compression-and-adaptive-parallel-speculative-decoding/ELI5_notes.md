# 31.1 A 14.08-to-135.69Token/s ReRAM-on-Logic Stacked Outlier-Free Large-Language-Model Accelerator with Block-Clustered Weight-Compression and Adaptive Parallel-Speculative-Decoding 通俗讲解

### 0. 整体创新点通俗解读

**痛点直击**

在边缘设备上跑大语言模型（LLM），最大的折磨是**自回归解码**带来的**外部内存访问**瓶颈。为了缓解这个问题，业界引入了**Speculative Decoding (SD)**：让一个小模型先猜一堆Token，大模型再并行验证。但在资源受限的边缘芯片上，SD不仅没跑快，反而处处添堵：
- **大模型量化掉精度**：为了省带宽，大模型必须做低比特量化，但激活值里的**Outliers**会让精度崩盘。用**Fast Walsh-Hadamard Transform (FWHT)**可以消除Outliers，但深层FWHT硬件实现太占面积。
- **小模型依然爆显存**：小模型虽然参数少，但边缘芯片的片上SRAM根本装不下，还得去外部DRAM里捞权重，带宽照样卡脖子。
- **瞎猜导致白忙活**：如果让小模型一次猜太多，大模型会拒绝超过90%的Token，算力全浪费在处理废稿上了。

**通俗比方**

把SD想象成一个**主厨（TLM）**和**学徒（DLM）**的配合。主厨做菜慢但准，学徒切菜快但可能切错。
- 以前的痛点是：主厨和学徒都要去很远的大仓库（**EMA**）拿食材，跑腿时间比做菜还长。
- 这篇论文的解法是：
  - **对付主厨的食材结块**：与其造个巨大的工业搅拌机（深层FWHT）来打散结块，不如用两个小号手持搅拌机（浅层FWHT）从不同角度搅，效果差不多但省钱省地。
  - **给学徒配个随身百宝箱**：把学徒需要的食材压缩成极简的调料包（**BVQ码本**），直接塞进他随身背的ReRAM背包里（**3D堆叠**），再也不用跑仓库了。
  - **动态调整切菜策略**：学徒先切一盘菜试试，如果主厨全盘接受，学徒就开启并行模式疯狂切菜；如果主厨不满意，立马退回单盘模式。同时，厨房里安排了一个**大堂经理（WDOS）**，专门统筹主厨和学徒的动线，防止他们抢锅抢菜刀。

**关键一招**

作者并没有在算法层面去死磕更复杂的量化或训练策略，而是通过**软硬件协同**，在硬件流水线里插入了三个精巧的"扭转"机制：
- **局部旋转替代全局旋转**：在TLM量化前，作者没有堆叠深层FWHT阵列，而是将非2的幂次维度分解，用**重叠的上下两级浅层FWHT**来近似全局旋转。这一招直接砍掉了92.7%的面积开销，实现了无Outlier的W4A8量化。
- **用ReRAM堆叠承载向量量化**：针对DLM，作者把权重通过**Blockwise Vector Quantization (BVQ)**聚合成码本，存入逻辑芯片上方的**ReRAM层**。通过高带宽的面对面Bumping接口，DLM的权重读取彻底摆脱了外部DRAM。
- **自适应并行验证与乱序调度**：在SD流程中，作者引入了**Adaptive Parallel SD (APSD)**。它根据TLM的验证反馈，动态在"短草稿串行"和"长草稿并行"之间切换。配合**Workload-Decoupled Out-of-order Scheduler (WDOS)**，将指令拆分到四个独立队列中乱序执行，榨干了片内并行的每一滴带宽。

### 1. Local Rotation Unit (LRU) with Decomposed FWHT

**痛点直击**

在 LLM 的低比特量化（如 W4A8）中，**Activation Outliers**（激活值异常点）是毁掉精度的罪魁祸首。少数几个极大的异常值会撑大整个量化 Scale，导致正常数值被压缩得面目全非。

- 业界已知 **FWHT**（快速沃尔什-哈达玛变换）能通过“旋转”特征空间，把异常值均摊到所有维度上，实现无损精度的 **Outlier-Free** 量化。
- 但在硬件上落地 FWHT 极其痛苦：LLM 的隐藏维度往往极大且非 2 的幂次（npot，如 LLaMA3-8B 的 14336）。
- 为了处理这种大维度，传统做法需要构建极深的级联 FWHT 阵列（深度高达 9）。
- 这导致硬件面积失控——一个完整的全局旋转阵列面积是 **4K INT8 MAC 阵列的 4.37 倍**。为了量化省点内存，结果控制电路比计算阵列还大，这是典型的得不偿失。

---

**通俗比方**

想象你要把一个长达 14336 米、高低不平的崎岖地表（包含几个突兀的山峰，即异常值）彻底推平。

- **全局旋转**：你调来一台超级巨大的 9 层楼高推土机，一口气从左推到右。地表确实平了，但这台机器太贵、太占地方（硬件面积爆炸）。
- **LRU 局部旋转**：你放弃造巨兽，改用两台便宜的小型 6 层推土机。第一台先推平上半区，第二台再推平下半区，并且让它们在交界处“重叠”作业一段距离。
- 虽然没有超级推土机推得那么绝对平整，但山峰已经被有效削平（异常值被消除），满足了通行的最低要求（满足 W4A8 量化精度），而你的设备成本（芯片面积）直接省了 92.7%。

---

**关键一招**

作者没有死磕如何缩小深层 FWHT 阵列，而是巧妙地在数学映射上做了**降维与拆分**，用“局部重叠”近似“全局计算”。

![](images/819f1860c7b38c47f1101cea72e371d35429b04af70f40b0620a4e01293f787f.jpg)

- **维度因式分解**：将非 2 的幂次维度 $n$ 分解为 $2^k \times m$。不再强求一次性处理整个长序列。
- **深度截断与重叠**：把原本深度为 9 的深层 FWHT，硬生生截断拆解为两个深度仅为 6 的低成本 FWHT（分别处理 upper 和 lower 特征）。
- **两阶段流水线**：
  - **RFA (Reconfigurable FWHT Array)**：可重构阵列，支持 $2^1$ 到 $2^6$ 的灵活变换，早期阶段合并相邻 FWHT 共享输入，省下昂贵的高精度加法器。
  - **HAU (Hadamard Accumulator Unit)**：把 npot Hadamard 矩阵的二元部分与 FP16 动态量化 Scale 融合，实现无 MAC 运算的累加。
- **结果**：通过这种“分而治之+局部重叠”的近似转换，LRU 成功在极小的面积开销下去除了异常值。

| 优化方案 | 硬件阵列深度 | 面积开销 | 量化效果 |
| :--- | :--- | :--- | :--- |
| 传统全局旋转 | 9 | 4.37× (相对 4K INT8 MAC) | 精确去异常 |
| **本文 LRU** | **6 (两段重叠)** | **0.32× (省 92.7%)** | **近似去异常** |

作者通过这种“不追求绝对完美，只追求够用”的工程妥协，成功在极其紧俏的边缘加速器面积预算内，塞进了一个能支撑 W4A8 高精度量化的旋转预处理单元。

### 2. ReRAM-stacked Process-Near-Memory (RS-PNM) with Blockwise Vector Quantization (BVQ)

**痛点直击**

在 Speculative Decoding (SD) 场景中，Draft LLM (DLM) 本应是“轻装上阵”的辅助角色。但在资源受限的边缘设备上，情况却极其尴尬：
- **片上内存装不下**：即便是百兆级别的 DLM，其权重也远超常规边缘芯片的 SRAM 容量。
- **频繁访存拖垮性能**：装不下就只能去 DRAM 里反复搬运权重，导致严重的 External Memory Access (EMA) 延迟。
- **带宽受限**：边缘设备的 DRAM 带宽极其有限，DLM 的频繁取指不仅自己慢，还挤占了 Target LLM (TLM) 的带宽资源。

辅助验证的 DLM 反而成了拖累整体推理速度的“猪队友”。

---

**通俗比方**

假设你是个大厨（Logic Die），要在极短时间内做出几百道菜（生成 Token）。
- **传统做法**：你的案板（SRAM）太小，放不下所有食材。每次做菜都要跑去远处的仓库（DRAM）拿食材，跑腿时间比炒菜还长。
- **RS-PNM 做法**：你在案板正上方搭了个密集的吊柜（堆叠 ReRAM），伸手就能拿到，带宽极高。
- **BVQ 做法**：吊柜空间也有限，不能放散装食材。于是你把食材按“菜系”提前打包成“半成品料理包”。吊柜里只存这些料理包，做菜时拿下来在案板上快速复原即可。

这里的“料理包”就是 Block-level Codebooks (CBs)，“吊柜”就是 ReRAM，“伸手拿”就是高带宽堆叠接口。

![](images/3b0e66f5354f2e35897d0d77f0d4a63bb57d425d98a22069e68b01a57a98d1b7.jpg) *Figure 31.1.4: ReRAM-stacked processing-near-memory (RS-PNM) architecture with blockwise vector quantization (BVQ) to avoid draft LLM external memory access (EMA).*

---

**关键一招**

作者并没有去死磕如何扩大 SRAM，而是通过**算法与硬件的垂直协同**，把权重存储和读取的逻辑彻底重构了：
- **算法端 (BVQ)**：不再逐个存储量化后的权重，而是将 DLM 权重按 Block 聚类，学习出 Block-level Codebooks (CBs)。原本庞大的权重矩阵被压缩成了极小的 CB 索引和码本。
- **硬件端 (RS-PNM)**：利用 bumping-based face-to-face 堆叠技术，把高密度 ReRAM 直接盖在 Logic Die 上方。CBs 全部存入 ReRAM，通过 2048 个 bump 提供超高带宽。
- **巧妙的防拥堵机制**：
  - 采用 **Vertical CB Mapping** 替代水平映射，规避了时钟频率限制导致的数据拥堵。
  - 设计 **Tile Fusion Unit (TFU)**：当多个 Token Tile 共享同一个 CB entry 时，TFU 会将它们融合，确保每个 CB 只从 ReRAM 中读取一次，直接将 CB 读取延迟减半。

**核心硬件参数对比**

| 参数 | 规格 |
| :--- | :--- |
| **堆叠技术** | Bumping-based face-to-face ReRAM-on-logic |
| **ReRAM 容量** | 8MB (单芯片) / 32MB (4芯片系统) |
| **堆叠带宽** | 25.6 GB/s @ 100MHz |
| **Bump 数量** | 2048 (Data) + 864 (Ctrl) |
| **ReRAM 功耗** | 49.54 mW / Die |

### 3. Tile Fusion Unit (TFU)

**痛点直击**

在这个 ReRAM-on-Logic 架构中，DLM 的权重被压缩成了 Codebook (CB) 存在 ReRAM 里。为了配合 ReRAM 的物理 bank 宽度限制，系统采用了 vertical CB mapping。但这带来了一个极度难受的副作用：**冗余访问**。
- 当你在逻辑层重构完整权重时，不同的 token tiles 可能会指向同一个 CB entry。
- 在 vertical mapping 下，系统像个没头苍蝇，遇到一个 tile 就去 ReRAM 里把这个 CB entry 抓出来一次。
- 结果就是：同一个 CB entry 被反反复复读取了无数次。这不仅把宝贵的 ReRAM 带宽挤爆了，还让 CB read latency 成了拖慢整个系统的累赘。

---

**通俗比方**

想象你在后厨做菜（重构权重），菜谱被拆成一张张卡片放在柜子里。
- 传统做法：你拿一个土豆去柜子找“炒土豆丝”的菜谱，看一眼；转头拿另一个土豆，又去柜子找同一张“炒土豆丝”的菜谱。你跑断腿，柜子门也被你开烂了。
- TFU 就像一个聪明的帮厨：他看你手里有一堆土豆，直接把它们**打包成一个篮子**，然后去柜子里只拿一次“炒土豆丝”的菜谱，回来给这一整篮土豆一起用。
- 这就是**数据复用**的极致体现：与其按需索取导致重复劳动，不如先按需聚类，一次性搞定。

---

**关键一招**

作者并没有去魔改 ReRAM 的硬件结构，也没有引入复杂的缓存层级，而是巧妙地在数据流经的路径上插了一个**逻辑合并层**。
- 具体操作：在从 ReRAM 加载 CB 之前，TFU 会先扫描即将处理的 token tiles。
- 核心扭转：它把那些**共享同一个 CB entry 的 token tiles 直接融合在一起**。
- 效果：系统在向 ReRAM 发请求时，相同的 CB entry 只会被 fetch 一次。这招“去重”直接把 CB read latency 砍掉了一半，让 ReRAM 的带宽利用率拉满。

![](images/3b0e66f5354f2e35897d0d77f0d4a63bb57d425d98a22069e68b01a57a98d1b7.jpg) *Figure 31.1.4: ReRAM-stacked processing-near-memory (RS-PNM) architecture with blockwise vector quantization (BVQ) to avoid draft LLM external memory access (EMA).*

| 优化策略 | CB Access 次数 | Latency 影响 | 带宽利用率 |
| :--- | :--- | :--- | :--- |
| 无 TFU (Vertical Mapping) | 冗余极高 | 严重拖累系统 | 低 |
| 有 TFU (Tile Fusion) | 每个 CB 仅 1 次 | **Latency 减半** | 高 |

### 4. Adaptive Parallel Speculative Decoding (APSD)

**痛点直击**

在 Speculative Decoding (SD) 场景中，大模型 (TLM) 负责验证，小模型 (DLM) 负责猜词。这看似美好，但在资源受限的边缘设备上极其难受：

- **短 Draft Length (DL) 的无奈**：猜得少，虽然 TLM 接受率高，但 TLM 频繁被唤醒去验证，外部存储访问 (EMA) 开销根本降不下来。
- **长 DL 的灾难**：让 DLM 猜长一点？一旦猜错，TLM 拒绝率飙升（超过 90% 的草稿 Token 被毙掉），DLM 纯属白费力气。
- **并行 SD 的死穴**：为了提速搞并行，让 DLM 和 TLM 同时干活。但 TLM 验证慢，DLM 很快就干完没事了，导致 DLM 严重闲置，ReRAM 带宽白白浪费。

| 策略 | 拒绝率 | Token 产出 | 核心痛点 |
| --- | --- | --- | --- |
| 短 DL | 低 | 低 | TLM 频繁唤醒，EMA 开销大 |
| 长 DL | 极高 (>90%) | 理论高 | DLM 算力浪费严重 |
| 并行 SD | 中 | 中 | DLM 闲置，ReRAM 带宽浪费 |

---

**通俗比方**

把 TLM 想象成一个极其严苛的老板，DLM 是个急于表现的实习生。

- **传统模式**：实习生写一句，老板审一句。安全，但老板被频繁打断，累得半死。
- **长草稿模式**：实习生一口气写十句。老板一看第一句就错了，后面九句全撕了。实习生白干，老板也觉得烦。
- **并行模式**：实习生在老板审阅时，跑到旁边房间继续写下一封。但老板审得太慢，实习生写完只能干等，甚至占着资源不让别人用。
- **APSD 模式**：实习生变聪明了，他会**看老板脸色行事**。先试探性写一句，如果老板全盘接受，且老板自己写的下一个词和实习生并行的首词对上了，说明“手感”对了，立刻切到并行狂写模式；一旦老板皱眉拒绝，立马丢弃草稿，退回安全的一句一句写。这样既不惹老板生气，又能在顺风局时最大化产出，绝不干等。

---

**关键一招**

作者并没有重新设计 DLM 或 TLM 的网络结构，而是巧妙地在**调度策略**上动刀，插入了一个**动态反馈机制**和**乱序调度器**：

- **动态切换策略**：
  - 起步阶段：采用非并行的短 DL 起草，保证低拒绝率。
  - 触发条件：如果 TLM **全盘接受**之前的草稿，且 TLM 最新生成的 Token 与并发 DLM 起草的第一个 Token **完全匹配**。
  - 状态维持：满足触发条件，继续保持并行 draft-and-verify 模式，榨干长 DL 的高产出红利。
  - 状态回退：一旦不匹配，立刻丢弃草稿 Token，退回非并行模式，止损。
- **Workload-Decoupled Out-of-Order Scheduler (WDOS)**：
  - 为了支撑这种动态并行，作者把工作负载解耦成 4 个独立的指令队列：**Inter-chip transceiver**、**Compute**、**ReRAM load** 和 **EMAC**。
  - 队列内部提取依赖标记，队列间通过同步计数器矩阵追踪状态。
  - 父队列就绪则发射指令，并通知子队列，彻底消除并行 draft-and-verify 时的资源竞争。

### 5. Workload-Decoupled Out-of-Order Scheduler (WDOS)

**痛点直击**

在 Adaptive Parallel Speculative Decoding (APSD) 场景下，DLM (Draft LLM) 生成草稿快，TLM (Target LLM) 验证慢。
- 如果按部就班地排队执行，TLM 在验证时，DLM 只能干等，导致 **ReRAM 带宽和计算单元大量闲置**。
- 如果强行让两者并行，DLM 和 TLM 又会**抢夺片内内存接口和计算资源**，造成拥堵，反而拖慢整体速度。
- 这种“顾头不顾尾”的资源竞争，让边缘加速器的并行解码效率大打折扣。

---

**通俗比方**

就像餐厅后厨的“切菜工”（DLM）和“炒菜工”（TLM）。
- 切菜工手脚麻利，炒菜工慢工出细活。
- 传统做法是切一盘炒一盘，切菜工大部分时间在发呆。
- 强行并行是两人同时开工，但厨房只有一个水龙头和砧板，两人抢来抢去，效率极低。
- WDOS 的做法是：把工作拆解成“拿菜”、“洗菜”、“切菜”、“炒菜”四个独立的任务池。谁手头没活了，就去任务池里看有没有自己能干的、且前置条件已满足的活。比如炒菜工还在炒上一盘，切菜工发现“拿菜”和“洗菜”的活没人干，就立刻去干，绝不闲着，也不跟炒菜工抢锅。

---

**关键一招**

作者并没有重头设计复杂的硬件控制逻辑，而是巧妙地在指令调度层插了一个**任务解耦与乱序执行**机制。
- 将 APSD 的工作负载拆解为 **4 个并行指令队列**：

| 指令队列 | 功能 |
| --- | --- |
| inter-chip transceiver | 片间收发 |
| compute | 计算 |
| ReRAM load | ReRAM加载 |
| EMAC | 外部内存访问控制 |

- 调度器不再死板地按顺序发指令，而是采用**依赖感知的乱序调度**。
- 通过维护一个**同步计数器矩阵**，追踪每个指令的依赖关系。只要前置指令完成，后续指令立刻发射，无需等待整个大任务完成。
- 这样既避免了 DLM 和 TLM 抢资源，又让硬件单元始终处于满载状态。
